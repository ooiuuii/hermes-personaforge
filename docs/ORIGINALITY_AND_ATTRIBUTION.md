# Originality And Attribution

PersonaForge combines an original Hermes-style business workflow with compatibility ideas from existing character-agent tooling.

## Original Work In This Repo

The following parts are original to this submission:

- paid persona-pack business flow,
- Qiance commerce entitlement mapping,
- redacted payment reconciliation proof,
- delivery margin gate,
- persona license manifest schema,
- runtime launch trace schema,
- safety redaction report,
- persona audit ledger,
- proof-manifest verification flow,
- React/Vite judging workbench,
- cinematic demo scenes and recording scripts,
- hackathon submission copy and documentation.

These pieces are what make the project a business-agent submission rather than only a character demo.

## Compatible Or Inspired Surfaces

The video demo uses the idea of a Shinsekai-compatible local character runtime: a visual companion can read screen context, react with voice, and switch expression routes. PersonaForge treats that runtime as an integration target, not as code owned by this repo.

The project is also inspired by common AI companion UX patterns:

- browser or page-reading interactions,
- voice responses,
- character expression changes,
- local runtime logs,
- importable persona or character manifests.

Those patterns are used to demonstrate the product experience after the Hermes business flow unlocks access.

## Not Redistributed Here

This repository does not include:

- Shinsekai runtime source code,
- third-party character packs,
- anime/game character sprites,
- voice checkpoints or samples,
- model weights,
- private OAuth/API credentials,
- payment secrets,
- raw customer payment data.

If a local demo uses private character assets, those assets remain outside Git and outside the downloadable public proof artifacts.

## Relationship To Hermes

PersonaForge is meant to become a Hermes plugin or skill surface. In the current hackathon version, Hermes-style logic is represented by deterministic scripts and proof artifacts:

- payment reconciliation,
- cost/margin decisioning,
- manifest unlock,
- runtime trace,
- safety proof,
- audit ledger.

The local companion runtime exists only to make the post-purchase experience visible in the video.

## Review Boundary

Judges and reviewers should evaluate the repo on:

- whether a paid order can be turned into an entitlement,
- whether the agent decides when to unlock or hold delivery,
- whether the runtime action is traceable,
- whether unsafe assets and secrets are kept out of the repo,
- whether the proof pack can be regenerated.

They should not treat the repo as a redistribution of any specific third-party character runtime or character IP.
